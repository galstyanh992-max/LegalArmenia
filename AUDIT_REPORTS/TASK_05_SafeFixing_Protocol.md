## Принцип: ZERO BLIND FIXES

Ни одно исправление не считается завершённым, пока не выполнен полный цикл верификации.

---

## Обязательный цикл для каждого исправления

```
НАЙТИ → ИЗОЛИРОВАТЬ → ИСПРАВИТЬ → ВЕРИФИЦИРОВАТЬ → УБЕДИТЬСЯ В ЦЕЛОСТНОСТИ → ПЕРЕЙТИ ДАЛЬШЕ
```

### Шаг 1: НАЙТИ
- Точно локализовать проблему: файл, строка, функция, таблица.
- Определить root cause, а не симптом.
- Проверить: не является ли проблема следствием другой более глубокой.

### Шаг 2: ИЗОЛИРОВАТЬ
- Определить все места, где проявляется та же проблема.
- Проверить все зависимые компоненты, которые могут быть затронуты исправлением.
- Составить список файлов под изменение.

### Шаг 3: ИСПРАВИТЬ
- Применить минимальное изменение, решающее проблему.
- Не вносить попутных "улучшений" — один fix = одна проблема.
- Сохранить все измененные файлы.

### Шаг 4: ВЕРИФИЦИРОВАТЬ
- После каждого изменения прочитать изменённый файл и проверить корректность.
- Проверить, что TypeScript-типы не нарушены (`tsc --noEmit` если применимо).
- Проверить, что связанные функции/компоненты видят правильный контракт.

### Шаг 5: УБЕДИТЬСЯ В ЦЕЛОСТНОСТИ
- Проверить список зависимых компонентов из Шага 2.
- Убедиться, что ни один из них не сломан изменением.
- Для edge functions: проверить все импортирующие их функции.
- Для shared модулей: проверить все пользователи модуля.

### Шаг 6: ПЕРЕЙТИ ДАЛЬШЕ
- Зафиксировать результат в соответствующем разделе отчёта.
- Обновить статус бага в Bug Register.
- Только после этого переходить к следующему исправлению.

## Запрещённые паттерны

| НЕЛЬЗЯ | ВМЕСТО ЭТОГО |
|--------|-------------|
| Исправлять несколько несвязанных проблем одновременно | Один fix за раз |
| Переходить к следующей проблеме, не проверив предыдущую | Верифицировать каждый fix |
| Вносить "косметические" изменения попутно | Строго по задаче |
| Изменять shared-модули без проверки всех зависимостей | Проверять всех потребителей |
| Писать изменение в DB/migration без проверки RLS-политик | Проверять RLS при каждом schema change |
| Изменять auth-логику без проверки на нескольких ролях | Тестировать на admin, lawyer, client, auditor |

## Приоритизация исправлений

Порядок, в котором фиксы должны выполняться:

```
CRITICAL (блокирует работу или безопасность)
    ↓
HIGH (нарушает ключевую функциональность или мониторинг)
    ↓
MEDIUM (снижает надёжность или усиливает риски)
    ↓
LOW (улучшения и чистота)
```

## Протокол для DB/Migration изменений

1. НИКОГДА не применять migration без просмотра текущего состояния таблицы.
2. Проверить все RLS-политики, которые затрагивает изменение.
3. Проверить FK-связи — не нарушит ли изменение cascade.
4. Для изменения ENUM: проверить все места, где enum используется в коде.
5. Для DROP: убедиться, что колонка/таблица не используется нигде.

---

## Протокол для isправлений в auth/security

1. Тестировать на КАЖДОЙ роли: admin, lawyer, client, auditor.
2. Тестировать unauthenticated запрос — должен получить 401/403.
3. Проверить, что смежные RLS-политики не открывают доступ в обход.
