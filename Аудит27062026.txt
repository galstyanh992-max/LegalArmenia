## Part 2: Architecture + Control Flow + Data Flow

### 1. Logical Architecture & Execution Architecture

Система построена на модульной, serverless архитектуре с использованием Supabase (PostgreSQL, Auth, Edge Functions) и внешних AI-провайдеров (через единый шлюз).

**ASCII Architecture Diagram:**
```text
[ React UI / Client ] 
        │ (JWT, REST)
        ▼
[ Supabase API Gateway ] ──────┐
        │                      │
        ▼                      ▼
[ Edge Functions ]       [ PostgreSQL (RLS) ]
  ├─ legal-chat            ├─ auth.users
  ├─ ai-analyze            ├─ cases, case_files
  ├─ kb-unified-search     ├─ knowledge_base (pgvector)
  └─ ... (48 others)       └─ audit_logs
        │
        ▼ (Sanitized prompts + RAG context)
[ AI Gateway Bypass (_shared/gateway-bypass.ts) ]
        │
        ▼
[ OpenAI / OpenRouter API (gemini-2.5-flash / embeddings) ]
```

### 2. Control Flow & Trust Boundaries

**VERIFIED Execution Logic (на основе `legal-chat` и `ai-analyze`):**
1. **Auth & Identity:** Каждая функция жестко проверяет JWT через `sb.auth.getUser()`. Доступ к данным ограничивается как на уровне кода (в `ai-analyze` явно проверяется `case_id` через authClient), так и на уровне базы данных (RLS).
2. **Input Validation & Safety:**
   - Строгие лимиты на длину (`message` до 5000 символов, история до 30 сообщений).
   - Внедрен слой `prompt-armor.ts` (`sanitizeUserInput`). Попытки инъекций логируются в `audit_logs`.
3. **Data Retrieval (RAG):**
   - В `ai-analyze` реализован **гибридный RAG**: сначала извлекаются точные ссылки на нормы (anchors) через `extractNormRefs`, затем выполняется семантический поиск (`dualSearch`), после чего результаты объединяются и сортируются по юридической иерархии (`authorityRank` — Конституция > ЕКПЧ > Кодексы > Прецеденты).
4. **Context Assembly:** Документы извлекаются из БД (OCR, Audio) и, если они слишком большие, пропускаются через `mapReduceSummarize` во избежание переполнения контекстного окна (Token Exhaustion Attack).
5. **LLM Invocation:** Запросы отправляются через внутренний роутер `openai-router.ts`. В `legal-chat` используется `callStreamBypass` с точным подсчетом сгенерированных токенов для биллинга (cost accounting).

**Trust Boundaries:**
- **Client -> Edge Function:** Недоверенная зона. Все входы санируются.
- **Edge Function -> PostgreSQL:** Доверенная зона. Вызовы идут через service_role для поиска по базе знаний (публичные данные) и через authClient для пользовательских данных (кейсов).
- **LLM -> Edge Function:** LLM рассматривается как полу-доверенная. Выдача транслируется напрямую клиенту (в `legal-chat` через streaming), что создает риск передачи галлюцинаций.

**Single Points of Failure:**
- Зависимость от `openai-router.ts`: сбой или неверная настройка ключей OpenAI/OpenRouter полностью останавливает систему.
- `search_legal_corpus_dual` RPC в PostgreSQL: медленный векторный поиск может вызывать таймауты Edge-функций (лимит времени Deno Deploy).

## Part 3: Function-by-Function Audit

### Function Audit: `legal-chat`
#### 1. Classification
- **Status:** VERIFIED (исходный код загружен и проанализирован)
- **Criticality:** High
- **Exposure:** User-facing

#### 2. Purpose
- **Declared:** Юридический чат-бот для ответов на вопросы по праву Армении с использованием базы знаний.
- **Verified:** Выполняет заявленное, используя RAG (база знаний + практика) и потоковую генерацию.

#### 3. Trigger and Entry Conditions
- HTTP POST с JWT-токеном. Проверяет наличие `message` и (опционально) `conversationHistory`.

#### 4. Inputs
- Строго валидирует длину `message` (<= 5000 chars) и ограничивает `conversationHistory` (<= 30 элементов, каждый <= 5000 chars).
- Элементы истории от пользователя дополнительно прогоняются через `prompt-armor.ts`.

#### 5. Internal Execution Logic
1. Auth check.
2. Rate limits check (вызов `checkRateLimits`).
3. RAG-поиск в `knowledge_base` и `legal_practice_kb` через shared-модуль.
4. Token budgeting (`applyBudgets`).
5. Prompt sanitization и логирование инъекций.
6. Вызов `callStreamBypass` и трансляция SSE-потока клиенту с параллельным подсчетом токенов.

#### 6. Failure Behavior
- Если поиск падает, ошибка логируется, но чат продолжает работу без контекста (Silent degradation).
- Если API провайдера возвращает 429 или 402, клиенту отдается внятная ошибка на армянском языке.

#### 7. Security / Safety / Privacy Risks
- **Risk:** Indirect Prompt Injection через базу знаний (Poisoned RAG). Если злоумышленник сможет загрузить документ в базу знаний (или если база содержит вредоносный текст), он попадет в контекст без санитарии.
- **Confidence:** High. Санитария применяется только к `USER_MESSAGE` и истории, но не к `kbContext`.

### Function Audit: `ai-analyze`
#### 1. Classification
- **Status:** VERIFIED
- **Criticality:** Critical
- **Exposure:** User-facing / Internal-hybrid

#### 2. Purpose
- **Declared:** Многоролевой анализ дел (Адвокат, Прокурор, Судья).
- **Verified:** Поддерживает 15 типов анализа. Выполняет сложный пайплайн: извлечение файлов -> Map-Reduce OCR/Audio -> извлечение юридических якорей -> RAG -> сборка контекста -> LLM вызов.

#### 3. Inputs
- `caseId`, `role` (15+ валидных ролей), опционально тексты законов, `strict_temporal` флаг.

#### 4. Internal Execution Logic
1. **Access Check:** Строгий чек доступа к `caseId` от имени текущего юзера (`authClient`).
2. **File Assembly:** Загрузка OCR-результатов и аудио-транскрипций. Для сырых файлов: прямая выкачка из Storage. DOCX парсится на лету. Изображения кодируются в Base64 для Vision-модели (лимит 5 шт / 5 МБ).
3. **Map-Reduce:** Крайне зрелый подход — длинные тексты сжимаются через `mapReduceSummarize`.
4. **Anchor Extraction & RAG:** Сначала ищет точные статьи (Anchors), затем делает векторный поиск. Результаты проходят дедупликацию и сортируются по весу `authorityRank`.
5. **Execution:** Контекст передается в нужный промпт (PROMPT_REGISTRY) и отправляется в LLM.

#### 5. Failure Behavior
- Отсутствие `reference_date` логируется как audit event `temporal_reference_date_missing` и добавляет warning в промпт LLM (или блокирует выполнение, если `strict_temporal=true`). Это признак высокой зрелости логики (Temporal Context Awareness).

#### 6. Security / Safety / Privacy Risks
- **Risk:** Утечка PII. Если OCR документы содержат ПДН, они отправляются в OpenAI/OpenRouter. В импортах есть `redactPII`, но в коде `ai-analyze/index.ts` он **не применяется** к `filesText` перед отправкой в LLM.
- **Confidence:** High (код прямо отправляет `mrResult.summary` в `fullCaseText` без вызова `redactPII`).

**Prompt Architecture (VERIFIED):**
Стек промптов обладает крайне высокой степенью инженерной проработки. 
- **Role Stability:** Каждая из основных ролей (Advocate, Prosecutor, Judge) имеет свой изолированный жесткий промпт, переключающий режим LLM.
- **Data Grounding:** Промпты явно запрещают выдумывать прецеденты и требуют вставлять шаблонную фразу: `Անալոգ դատական պրակտիկա (KB): հասանելի չէ...`, если данных в RAG нет. Это сильный анти-галлюцинационный паттерн.
- **Temporal Strictness:** В промптах заложена обработка временных рамок (`reference_date`). Если дата неизвестна, система передает `temporalWarning` в контекст LLM, чтобы LLM знала о потенциальной неактуальности норм.

**Agent / Orchestration Audit (VERIFIED):**
- Агенты не являются автономными (loop-based). Система работает по паттерну **One-Shot RAG Execution**.
- **Orchestration:** Пайплайн жестко контролируется TypeScript-кодом (`ai-analyze`). Код сам собирает файлы, делает OCR Summarization (Map-Reduce), делает RAG поиск и только потом собирает финальный промпт. Это безопаснее, чем отдавать LLM инструменты (tool-calling) для самостоятельного поиска.
- **Tools:** LLM-модели (gemini-2.5-flash) в данной системе **не используют инструменты** (zero tool exposure). Все вызовы базы данных делает детерминированный бэкенд. Риск Tool Hijacking равен нулю (VERIFIED).

**Memory Audit (VERIFIED):**
- Постоянная память (Long-term memory) у LLM отсутствует.
- Диалоговая память ограничена 30 последними сообщениями (с жестким лимитом по токенам).
- Риск отравления памяти (Poisoning) купируется тем, что `conversationHistory` от пользователя перед каждой отправкой пропускается через санитайзер `prompt-armor.ts`.

  1. Извлечение точных якорей (`extractNormRefs`) из текста дела.
  2. Семантический поиск по 38,000+ эмбеддингам (`text-embedding-3-small`).
  3. Дедупликация и **ранжирование по иерархии права** (`authorityRank` — Конституция имеет больший вес, чем подзаконные акты).
- Лимиты: В контекст передается до 50 источников.

**Freshness & Integrity (CLAIMED):**
- Документация утверждает, что база содержит актуальное законодательство. Однако автоматический механизм обновления (Freshness Policy) не найден. Без регулярного парсинга Arlis.am база со временем устареет (Stale Index Risk).
